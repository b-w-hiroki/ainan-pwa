# AINAN 釣り×街おこし 実装計画書

- **前提**: PWA / 振動なし / バックエンド含むサービス全体 / アップデート・コンテンツ追加を前提
- **目標**: 触り心地をできる限り良くする

---

## 1. 全体アーキテクチャ

```
[PWA (SPA)]  ←→  [API Gateway / BFF]  ←→  [バックエンドAPI]
     │                    │                        │
     │                    │              ┌──────────┼──────────┐
     │                    │              │          │          │
     └─ Service Worker    └─ 認証/セッション    DB     ストレージ  ジョブキュー
        (オフラインキャッシュ)                    (マスタ/ユーザー/取引)
```

- **フロント**: PWA（SPA）、オフライン閲覧用キャッシュ、インストール可能
- **バックエンド**: REST or GraphQL API、認証・ポイント・景品交換・マスタ配信を担当
- **コンテンツ更新**: マスタはAPI経由で取得し、バージョン/日付でキャッシュ invalidation

---

## 2. PWA 設計

| 項目 | 方針 |
|------|------|
| **マニフェスト** | `display: standalone`, アイコン複数解像度, テーマ色・背景色（愛南ブランド） |
| **Service Worker** | Workbox 等で静的アセット・APIレスポンスのキャッシュ戦略（Network First for API, Cache First for static） |
| **オフライン** | 一覧・図鑑など読み取りはキャッシュから表示可能に。釣り実行・ポイント加算はオンライン必須 |
| **更新** | SW 更新時は「新しいバージョンあり」を通知し、リロードで適用。強制更新は API の `minClientVersion` で制御 |

## 3. 触り心地（UX・フィードバック）設計

振動なしで「気持ち良い感触」を出すための方針。

### 3.1 入力フィードバック

- **タッチ**: ボタン・スライダーは `:active` で軽いスケール（0.97）＋透明度変化。可能なら `touch-action` でスクロールと干渉させない。
- **キャスト・リール**: スワイプ/ドラッグに合わせた**連続的なビジュアル変化**（ゲージ・アニメーション）で操作量が分かるようにする。
- **あばれ制御**: 傾きに応じて矢印・ゲージが滑らかに追従（60fps 目標）。カクつき防止のため `requestAnimationFrame` とデバウンスのバランス。

### 3.2 画面演出

- **60fps 目標**: ゲーム画面は `will-change` やレイヤー分離で GPU レンダリング。重い DOM 操作は避け、Canvas または WebGL を検討。
- **マイクロインタラクション**: ヒット時・釣り上げ時は短いアニメ（0.2〜0.4s）、easing は `ease-out` 系で「きちんと止まる」感じに。
- **ローディング**: スケルトン or 軽いアニメで待ち時間を感じさせない。ポイント加算は数字カウントアップで見せる。

### 3.3 音

- **SE**: キャスト・ヒット・リール・成功/失敗を少ないキーサウンドで差別化。音量はユーザー設定で OFF 可能に。
- **BGM**: 場面ごと ON/OFF。再生は Web Audio API で遅延なく。

### 3.4 パフォーマンス

- **FCP / LCP**: 初期表示は必要最小の JS/CSS。フォントは `font-display: optional` または `swap` でブロックしない。
- **INP**: タッチ・クリックへの応答を 200ms 以内に。重い処理は Web Worker や idle 後に回す。

## 4. バックエンド設計

### 4.1 責務

- 認証・セッション
- ユーザー・釣果・ポイント・所持品
- マスタ配信（魚・竿・景品・施設・季節イベント）
- 景品交換申込・決済連携・配送情報
- 運用（コンテンツ追加・イベント開始/終了）

### 4.2 API 方針

- **REST** を基本。一覧・フィルタが多用されるマスタは `GET /api/v1/masters/...` で一括 or 種別取得。
- **バージョン**: `Accept` または `?v=` でマスタバージョン指定。クライアントは `lastModified` / `version` を保持し、変更時のみ再取得。
- **認証**: JWT（アクセス＋リフレッシュ）またはセッションCookie。PWA は Cookie も利用可能。

### 4.3 主要エンドポイント例

| 種別 | 例 | 備考 |
|------|-----|------|
| 認証 | `POST /auth/login`, `POST /auth/refresh` |  |
| ユーザー | `GET/PATCH /users/me`, `GET /users/me/points` |  |
| 釣り | `POST /fishing/cast`, `POST /fishing/result` | 結果でポイント加算 |
| マスタ | `GET /masters/fish`, `GET /masters/rods`, `GET /masters/rewards` | クエリで有効期間・地域フィルタ |
| 景品 | `GET /rewards`, `POST /rewards/exchange` | 交換は在庫・ポイントチェック後 |
| コンテンツ | `GET /content/events`, `GET /content/facilities` | 更新日でキャッシュ |

### 4.4 データベース（論理）

- **users**: id, 認証子, 表示名, ポイント残高, 設定, 作成/更新日時
- **fishing_logs**: ユーザー, 魚ID, ポイント, 日時（集計・ランキング用）
- **exchanges**: ユーザー, 景品ID, ポイント消費, 配送先, 状態, 日時
- **masters**: 魚・竿・景品・施設・イベントなどを**JSON or 正規化テーブル**で保持。`valid_from` / `valid_to` で期間管理
- **content_versions**: マスタまたはコンテンツ単位の `version` / `published_at`（キャッシュ invalidation 用）

### 4.5 コンテンツ・アップデート前提

- **マスタは「公開日・終了日」で管理**: 新魚・新景品・イベントを追加しても既存クライアントは古いキャッシュで動き、API が新しい `version` を返したら PWA が再取得。
- **フロントの強制更新**: API の `GET /config` やヘルスで `minClientVersion` を返し、それより古いクライアントは「アップデートしてください」と表示。
- **A/B や段階ロールアウト**: 景品の割引率や新機能を、ユーザーセグメント or フラグで切り替え可能にしておくと運用しやすい。

## 5. ディレクトリ・リポジトリ構成案

```
AINAN/
├── apps/
│   ├── web/                 # PWA (SPA)
│   │   ├── public/
│   │   │   ├── manifest.json
│   │   │   └── sw.js (or Workbox)
│   │   └── src/
│   └── api/                 # バックエンド API
│       ├── src/
│       │   ├── routes/
│       │   ├── services/
│       │   └── db/
│       └── prisma or migrations
├── packages/
│   ├── types/               # 共通型 (魚・景品・API 契約)
│   └── constants/           # 共通定数
├── docs/
│   └── IMPLEMENTATION_PLAN.md (本ドキュメント)
└── package.json             # モノレポ root
```

- マスタデータは `api` の seeds または管理画面から投入。コンテンツ追加は DB 更新＋必要に応じて `content_versions` を更新。

## 6. 実装フェーズ案

| フェーズ | 内容 |
|----------|------|
| **1. 基盤** | モノレポ・API スケルトン・認証・DB スキーマ・マスタ CRUD のたたき |
| **2. PWA 基盤** | SPA 構築, manifest, Service Worker, オフライン戦略, 触り心地の基盤（タッチスタイル・60fps 方針） |
| **3. コアゲーム** | 釣りフロー（キャスト→ヒット→リール→あばれ）の UI/API 実装、ポイント付与 |
| **4. マスタ連携** | 魚・竿・景品のマスタ取得とキャッシュ、バージョンによる更新 |
| **5. 景品交換** | 交換 API・在庫・決済連携（外部 Web 決済）、履歴表示 |
| **6. コンテンツ拡張** | イベント・施設・季節コンテンツの追加フローと表示 |
| **7. 運用** | minClientVersion、ログ・監視、コンテンツ追加手順のドキュメント化 |

## 7. 技術スタック例（参考）

- **PWA**: React or Vue + Vite, Workbox (vite-plugin-pwa)
- **API**: Node (Fastify/Express) or Go, Prisma or Drizzle
- **DB**: PostgreSQL（トランザクション・集計向き）
- **ストレージ**: 景品画像等は S3 互換 + CDN
- **認証**: JWT + リフレッシュトークン or セッションストア

この計画に沿えば、PWA・振動なし・バックエンド含むサービスを、アップデートとコンテンツ追加を前提に一貫して実装できます。触り心地はフロントの「入力→画面・音のフィードバック」を 60fps と短いアニメで統一すると効果が出ます。
